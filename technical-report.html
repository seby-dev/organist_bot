Executive Summary

What the project does and why it exists

OrganistBot is a personal automation tool built to solve a concrete, repeatable problem: monitoring a specialist job board (organistsonline.org) for organ-playing engagements that match a musician's requirements, notifying them by email when a suitable gig appears, and handling the administrative follow-through — application email and Google Calendar entry — with minimal manual effort.

Without automation, the workflow is tedious and time-sensitive: check the website several times a day, manually read each new listing, decide whether it meets fee and distance requirements, compose an application email, and diary the date. OrganistBot compresses all of that into a background process that polls every two minutes and acts in seconds.

High-level architecture

The system is structured as a three-phase pipeline that runs on a scheduler: Scrape (fetch and parse the listing page), Filter (apply a composable chain of business rules), and Notify (send emails and update the calendar). Two separate long-running processes operate simultaneously: the main scheduler and a unified Telegram bot. The Telegram bot routes all free-text messages to a single Claude-powered agent (unified_agent.py) that handles gig calendar management, invoice generation, filter configuration, pipeline stats, and runtime config — all through plain-English conversation.

Configuration is fully externalised to a .env file via pydantic-settings. Runtime-mutable filter settings (blacklist, unavailable periods, available-only periods) are stored in a file-backed JSON store (filter_store.py) that the scheduler reads fresh on every tick. Pipeline runtime overrides (min fee, max travel, poll interval) live in a second file-backed store (runtime_config_store.py) and take effect on the next polling tick. Each concern — HTTP transport, HTML parsing, business rules, email delivery, calendar integration, PDF generation, persistence, logging — lives in its own module with no circular dependencies.

After each pipeline tick, structured logs are flushed to Google Sheets via a custom logging.Handler subclass (sheets_logger.py), enabling pipeline stats queries and anomaly alerting directly through the Telegram bot. The codebase runs locally on macOS under launchd, ships with 484 pytest tests, a two-job GitHub Actions CI pipeline (lint/type-check + tests), and pre-commit hooks that catch formatting and type errors before every commit.

Architectural pattern: the pipeline

A pipeline is a sequence of stages where the output of each stage is the input to the next. OrganistBot uses this pattern because the problem is naturally sequential: you cannot filter what you have not scraped, and you cannot notify about gigs you have not filtered.

The key advantage over a monolithic approach (one big function that scrapes, filters, and notifies) is that each phase is independently testable, swappable, and logged. You can unit-test the filter chain with a list of mock Gig objects without ever touching the network. You can replace SMTP with a file-writer for testing. You can add a new filter by writing one class and one line in main.py.

💡 First Principles: Why not write everything in one file?

Imagine the entire bot as a single 500-line function. To test "does the fee filter work?", you'd need a real internet connection, a real email server, and a Google Calendar. Now imagine a colleague breaks the scraper — suddenly the filter tests fail too, even though the filter code is fine. You can't see which part broke. You can't test parts in isolation. You can't swap the email provider without rewriting half the function.

Separation of concerns — splitting the code into modules by responsibility — is the engineering equivalent of specialised tools. A screwdriver does one thing perfectly. When it breaks, you replace just the screwdriver, not the entire toolbox.

Data flow

Data enters the system as raw HTML bytes from an HTTP response. BeautifulSoup parses it into a tree of Python objects. extract_basic_details() walks that tree and extracts strings into a plain Python dict. That dict is unpacked into a Gig dataclass — a strongly-typed, immutable-ish record. The filter chain receives a list[Gig] and returns a filtered list[Gig]. The notifier receives that same list and produces side effects (emails, CSV writes). No global mutable state is shared between phases.

Every module in the package. It is the only file that knows about all the others.

260 lines

Code walkthrough

# The Scraper is constructed ONCE before the scheduler loop,
# then passed into every main() call. This keeps the TCP connection
# pool alive across runs — no new TLS handshake every 2 minutes.
scraper = Scraper()
try:
    main(scraper)  # run immediately on startup
    schedule.every(settings.poll_minutes).minutes.do(main, scraper)
    while True:
        try:
            schedule.run_pending()
        except Exception:
            logger.exception("Unhandled exception in scheduled run")
            _send_telegram_alert("❌ OrganistBot crashed — check logs.")
        time.sleep(1)
finally:
    scraper.session.close()

The while True loop ticks every second. schedule.run_pending() fires main(scraper) when the timer is due. The inner try/except catches any exception from a run and sends a Telegram alert — but does not re-raise it, so the scheduler continues. The finally block ensures the HTTP session is closed cleanly on keyboard interrupt.

# Load seen gigs ONCE before the loop, not inside it.
# On a busy listing page with 30 gigs, this avoids 30 file reads.
seen_gigs_set = load_seen_gigs() if settings.enable_seen_filter else set()

# Pre-filter: applied using only basic details (no detail page fetch needed).
pre_filter = GigFilterChain()
if settings.enable_fee_filter:
    pre_filter.add(FeeFilter(min_fee=settings.min_fee))
if settings.enable_sunday_time_filter:
    pre_filter.add(SundayTimeFilter())

# For each listing element on the page:
for gig_el in gigs_div:
    basic = scraper.extract_basic_details(gig_el)
    if settings.enable_seen_filter and basic.get("link") in seen_gigs_set:
        continue              # already notified — skip instantly
    if not pre_filter.is_valid(Gig(**basic)):
        continue              # cheap rejection — no HTTP request
    extra = scraper.extract_full_details(scraper.fetch(basic["link"]))
    gig_list.append(Gig(**{**basic, **extra}))

This is the core of the N+1 optimisation. A detail-page HTTP request is only made when all cheap checks pass. On a typical poll, nearly every gig is already seen, so the loop body usually exits at the first continue.

💡 First Principles: What is a scheduler?

A scheduler is a piece of code that runs other code at regular intervals. The simplest possible scheduler is while True: do_work(); time.sleep(120). The schedule library adds a declarative API on top of this — you describe what to run and when, and the library handles the timing math. The while True: schedule.run_pending(); sleep(1) pattern is "cooperative" — no threads, no OS timers, just a tight loop that checks whether anything is due.

💡 First Principles: What are environment variables?

Every running process on your computer inherits a dictionary of key/value pairs from the operating system, called environment variables. They are a standard Unix mechanism for passing configuration to programs without hard-coding values into source code. A .env file is simply a text file of key=value pairs that a library like python-dotenv loads into the process environment on startup. The crucial benefit is that secrets (API keys, passwords) live outside the codebase, so they are never accidentally committed to git.

💡 First Principles: What is a dataclass?

A dataclass is Python's way of saying "this class primarily exists to hold data". Before dataclasses (Python 3.7), you'd write __init__ yourself, assign every parameter to self.x, and write a __repr__ for debugging. A dataclass generates all of that from the class body annotations. It is not magic — you can inspect the generated code with import inspect; print(inspect.getsource(Gig.__init__)).

Retry logic with tenacity

@retry(
    retry=retry_if_exception_type(RequestException),
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=1, max=15),
    before_sleep=_log_retry,
)
def fetch(self, url: str) -> str:
    response = self.session.get(url, timeout=self.timeout)
    response.raise_for_status()
    return response.text

The @retry decorator wraps the function so that if it raises a RequestException, it waits (1s, then 2s, then 4s, capped at 15s) and tries again up to 3 times. The before_sleep hook logs each retry with the attempt number and wait time. This is the exponential backoff pattern — waiting progressively longer between retries to give an overloaded server time to recover.

Connection pooling

def __init__(self, ...):
    self.session = requests.Session()
    adapter = HTTPAdapter(pool_connections=1, pool_maxsize=10)
    self.session.mount("https://", adapter)
    self.session.mount("http://", adapter)

A requests.Session reuses TCP connections. Mounting an HTTPAdapter with pool_maxsize=10 tells the session to keep up to 10 connections in its pool. For a single-domain scraper, this means the TLS handshake (which can take 200–500ms) happens once per boot, not once per fetch.

Type-safe attribute extraction

def _extract_link(self, booking: Tag) -> str | None:
    anchor = booking.find("a", class_="noselect")
    if not anchor:
        return None
    try:
        return settings.base_url + str(anchor["href"])
    except (KeyError, TypeError):
        return None

@staticmethod
def _get_sibling_text(detail_element: Tag, label: str) -> str | None:
    element = detail_element.find("h3", string=label)  # type: ignore[call-overload]
    if element and element.find_next_sibling("p"):
        return element.find_next_sibling("p").get_text(strip=True)
    return None

anchor["href"] is typed as str | AttributeValueList in BeautifulSoup's stubs — wrapping it in str() makes the concatenation type-safe and works correctly for both cases. The _get_sibling_text pattern navigates the HTML DOM tree by finding a labelled heading and then looking for its next sibling paragraph — a common pattern for scraping structured detail pages.

💡 First Principles: What is web scraping?

Web scraping is the process of programmatically downloading web pages and extracting data from them. A web page is delivered as HTML — a structured text document. BeautifulSoup parses that text into a tree of Python objects (tags, strings, attributes) that you can navigate and query. The challenge is that websites are designed for humans, not programs: the structure can change without warning, fields may be missing, and the server may rate-limit or block automated requests.

💡 First Principles: What is exponential backoff?

When a network request fails, immediately retrying usually makes things worse — if the server is overloaded, hammering it with retries increases the load. Exponential backoff waits 1 second, then 2, then 4, then 8 — each wait doubling the previous. This gives the server time to recover and spreads out retry traffic across clients. It is the standard pattern used by AWS SDKs, Google Cloud SDKs, and most professional HTTP clients.

organist_bot/filters.py

The Filter Chain — Heart of the Business Logic

At 501 lines, this is the largest and most important module. It contains four parsing helpers and seven filter classes, all composed by GigFilterChain.

Parsing helpers (pure functions)

def parse_min_fee(fee_str: str | None) -> float | None:
    # "£80 - £120"  →  80.0  (take the minimum)
    # "£100+"       →  100.0
    # "Negotiable"  →  None
    if not fee_str: return None
    if re.search(r"neg|negotiable|expenses", fee_str, re.IGNORECASE):
        return None
    amounts = re.findall(r"£?\s*([0-9]+(?:\.[0-9]{1,2})?)", fee_str)
    numbers = [float(a) for a in amounts]
    return min(numbers) if numbers else None

These functions are pure — no side effects, no global state. Given the same input, they always return the same output. This makes them trivially testable: assert parse_min_fee("£80 - £120") == 80.0. The regex r"£?\s*([0-9]+(?:\.[0-9]{1,2})?)" matches optional £ sign, optional whitespace, then a decimal number.

Filter classes as callables

class FeeFilter:
    def __init__(self, min_fee: float, weekday_min_fee: float = 120):
        self.min_fee = min_fee
        self.weekday_min_fee = weekday_min_fee

    def __call__(self, gig: Gig) -> bool:
        weekday = parse_weekday(gig.date)
        is_weekday = weekday in (0, 1, 2, 3, 4)
        required = self.weekday_min_fee if is_weekday else self.min_fee
        amount = parse_min_fee(gig.fee)
        return amount is not None and amount >= required

Each filter is a callable object — a class with a __call__ method. This means you can pass it anywhere a function is expected (Callable[[Gig], bool]), but it can also hold state (like min_fee). A plain function couldn't remember its configuration.

The GigFilterChain

class GigFilterChain:
    def __init__(self):
        self._filters: list[Callable[[Any], bool]] = []

    def add(self, filter_fn) -> "GigFilterChain":
        self._filters.append(filter_fn)
        return self  # enables: chain.add(A).add(B).add(C)

    def is_valid(self, gig: Gig) -> bool:
        for f in self._filters:
            if not f(gig):       # short-circuit: stop at first failure
                logger.debug("Gig rejected", extra={"filter": repr(f), ...})
                return False
        return True

    def apply(self, gigs: list[Gig]) -> list[Gig]:
        return [gig for gig in gigs if self.is_valid(gig)]

The chain pattern embodies the Open/Closed Principle: you can add new filters without modifying the chain itself. The return self in add() enables a fluent interface. The short-circuit in is_valid means an expensive filter (like the Google Maps API call in PostcodeFilter) is only reached if all cheap filters pass first.

💡 First Principles: What is the __call__ protocol?

In Python, any object with a __call__ method can be called like a function: obj(args). This is Python's way of unifying callable things — functions, methods, and objects — under one interface. The type hint Callable[[Gig], bool] says "anything that accepts a Gig and returns a bool", which includes plain functions, lambda expressions, and callable objects. Using callable objects instead of plain functions lets you bundle configuration (like min_fee) with the logic that uses it, keeping related things together.

💡 First Principles: What is short-circuit evaluation?

Short-circuit evaluation means stopping as soon as the answer is known. In a chain of AND conditions, if the first is false, the rest cannot change the result — so there is no need to evaluate them. Python's and operator does this natively: False and expensive_call() never calls expensive_call(). The filter chain mimics this behaviour explicitly: once a filter returns False, the loop breaks and the remaining filters are skipped.

organist_bot/storage.py

Persistent Seen-Gig Store

A thin module around Python's csv module. load_seen_gigs() reads a CSV file and returns a set[str] of gig URLs that have already been notified. save_seen_gigs(seen) overwrites the file with the full set — callers must pass the union of previously-seen and newly-valid links (seen_gigs_set | {gig.link for gig in valid_gigs}) so historical entries are never lost. A set is used rather than a list because membership testing (url in seen_gigs) is O(1) for sets vs O(n) for lists.

def load_seen_gigs(filepath: str = "data/seen_gigs.csv") -> set[str]:
    path = Path(filepath)
    if not path.exists():
        return set()
    seen = set()
    with path.open(newline="") as fh:
        for row in csv.reader(fh):
            if row:
                seen.add(row[0])   # first column is the link
    return seen

# In main.py — always pass the union so historical entries survive:
save_seen_gigs(seen=seen_gigs_set | {gig.link for gig in valid_gigs})
💡 First Principles: Why a set, not a list?

A Python set stores its elements in a hash table. When you ask "is this element in the set?", Python computes the element's hash (a number) and looks directly at the corresponding bucket — O(1), constant time regardless of set size. A list stores elements sequentially. "Is this element in the list?" requires scanning every element — O(n), linear time. With 10,000 seen gigs, a list would do 10,000 comparisons per gig checked; a set does roughly 1.

organist_bot/logging_config.py

Dual-Handler Logging System

One of the most carefully engineered modules in the project. Sets up two log handlers on the root logger: a colour-coded ANSI console handler (INFO+) for human reading, and a rotating JSON file handler (DEBUG+) for machine parsing.

Run ID correlation

_run_id: contextvars.ContextVar[str] = contextvars.ContextVar("run_id", default="")

class RunIdFilter(logging.Filter):
    def filter(self, record: logging.LogRecord) -> bool:
        record.run_id = _run_id.get("")
        return True

Every log record in a run shares the same 8-character hex run_id. This means you can filter all log lines from a single poll run with jq 'select(.run_id == "a1b2c3d4")' gigs.log. The ContextVar is set once at the top of main() and automatically propagated to every log call within that execution context.

JSON formatter

class JSONFormatter(logging.Formatter):
    def format(self, record: logging.LogRecord) -> str:
        doc = {
            "timestamp": ...,  # ISO 8601 with milliseconds
            "run_id": ...,
            "level": record.levelname,
            "logger": record.name,
            "message": record.getMessage(),
        }
        # Merge all extra fields supplied by the caller
        for key, value in record.__dict__.items():
            if key not in _STDLIB_FIELDS:
                doc[key] = value
        return json.dumps(doc, default=str)

The extra={...} arguments passed to every logger.info() call appear as top-level keys in the JSON object. This structured approach means you can query logs like a database: jq 'select(.elapsed_ms > 1000)' gigs.log finds all slow runs.

Every external API call emits an elapsed_ms field in its log record: the Google Maps Distance Matrix query (in PostcodeFilter._query()), both Google Calendar operations (has_event_on_date and add_gig), each SMTP email dispatch (Notifier._dispatch()), and the Telegram crash alert. This gives per-call latency visibility across every integration in a single jq query.

Idempotency guard

if root.handlers:
    return   # already configured — don't add duplicate handlers

Without this guard, calling setup_logging() twice (e.g. in a test) would add a second set of handlers, causing every log line to appear twice. This is a common Python logging pitfall.

💡 First Principles: Why two log formats?

The human reading a terminal wants colour, alignment, and brevity. The script parsing log files at 3am wants a reliable, machine-readable structure. JSON satisfies the latter because every language has a JSON parser, the structure is predictable, and tools like jq, pandas, and Elasticsearch can query it. The ANSI console formatter prioritises readability — fixed-width columns and colour coding let your eye jump immediately to warnings and errors.

organist_bot/notifier.py

Email Notifier

Handles all outbound email. Uses a transport abstraction — SMTPTransport for production, FakeTransport for testing — so no real emails are sent during test runs.

class SMTPTransport:
    def send(self, sender, recipient, cc, msg):
        with smtplib.SMTP_SSL("smtp.gmail.com", 465) as server:
            server.login(sender, self._password)
            all_recipients = [recipient] + (cc or [])
            server.sendmail(sender, all_recipients, msg.as_string())

class FakeTransport:
    def __init__(self):
        self.sent: list[dict] = []
    def send(self, sender, recipient, cc, msg):
        self.sent.append({"to": recipient, "subject": msg["Subject"]})

Both classes share an implicit interface: they have a send(sender, recipient, cc, msg) method. This is duck typing — Python doesn't require a formal interface declaration. Any object with the right method will work.

💡 First Principles: What is a template engine?

A template engine separates what to say from how to format it. Jinja2 templates are HTML files with special placeholder syntax: {{ gig.fee }} gets replaced with the actual fee value at render time; {% for gig in gigs %} loops over a list. organist_bot/filter_store.py

Runtime Filter Store

A file-backed store for the three runtime-editable filter values: blacklisted emails, unavailable periods, and available-only periods. All state lives in data/filter_config.json and is read fresh on every call, so changes made via Telegram commands take effect on the next scheduler tick without a restart.

_CONFIG_FILE = Path("data/filter_config.json")
_DEFAULTS: dict[str, list[str]] = {
    "blacklist_emails": [],
    "unavailable_periods": [],
    "available_only_periods": [],
}

def _read() -> dict[str, list[str]]:
    if not _CONFIG_FILE.exists():
        return dict(_DEFAULTS)
    with _CONFIG_FILE.open() as f:
        return {**_DEFAULTS, **json.load(f)}

def _write(data: dict[str, list[str]]) -> None:
    _CONFIG_FILE.parent.mkdir(parents=True, exist_ok=True)
    with _CONFIG_FILE.open("w") as f:
        json.dump(data, f, indent=2)

The public API exposes six functions: blacklist_emails(), unavailable_periods(), available_only_periods() (readers), and add_blacklist_email(), remove_blacklist_email(), add_period(key, period), remove_period(key, period) (writers). Writers return a boolean indicating whether the store actually changed — the Telegram bot uses this to reply "Already listed" vs "Added ✓". The store is not responsible for calendar side-effects; the unified agent layer calls GoogleCalendarClient.block_period / unblock_period after a successful write.

💡 First Principles: Why read fresh on every call?

The scheduler and the Telegram bot run as separate OS processes. Sharing state via an in-memory singleton would require inter-process communication (a socket, a queue, a database). Reading the JSON file fresh on every scheduler tick is simpler: the file system is the IPC channel. The cost — a few microseconds per read — is negligible compared to the HTTP requests in the same tick. This is the Unix philosophy: use the file system when it's sufficient.

organist_bot/integrations/sheets_logger.py

Google Sheets Logger

A logging.Handler subclass that buffers structured log records in memory and appends them to a Google Sheets spreadsheet in batch after each pipeline tick. This gives the bot a queryable observability store without needing a database.

class SheetsLogger(logging.Handler):
    def emit(self, record: logging.LogRecord) -> None:
        self._buffer.append(self._format_row(record))  # buffer in memory

    def flush_to_sheets(self) -> None:
        if not self._buffer:
            return
        self._service.spreadsheets().values().append(
            spreadsheetId=self._spreadsheet_id,
            range="Sheet1!A1",
            valueInputOption="RAW",
            body={"values": self._buffer},
        ).execute()
        self._buffer.clear()

Buffering is the key design decision: appending one row per log record would make 30+ API calls per tick. Flushing once per tick reduces that to a single batch append regardless of how many records were emitted. The handler is attached to the root logger in main.py after the pipeline run completes.

The query_run_stats method reads back recent rows, parses the structured details JSON column, and aggregates pipeline stats — total runs, gigs listed/pre-filtered/valid, filter rejection breakdown — used by the get_gig_stats tool in the unified agent.

💡 First Principles: Why subclass logging.Handler?

Python's logging framework is designed around handlers — objects that decide where log records go. The standard library ships handlers for files, streams, HTTP, email, and syslog. By subclassing logging.Handler, you plug into the same system without changing any of the existing logger.info() call sites. The entire pipeline is already instrumented; the Sheets handler just adds a new destination for the records it was already producing.

Runtime Config Store

A file-backed store for pipeline runtime overrides, following the same read-fresh-on-every-call pattern as filter_store.py. The scheduler reads overrides on each tick; the unified agent writes them via the manage_config tool. Supports three keys: min_fee, max_travel_minutes, and poll_minutes.

class RuntimeConfigStore:
    def get(self, key: str, default: int) -> int:
        return _read().get(key, default)   # .env default used if no override

    def set(self, key: str, value: int) -> None:
        data = _read(); data[key] = value; _write(data)

    def reset(self, key: str) -> bool:
        data = _read()
        if key not in data: return False
        del data[key]; _write(data); return True

runtime_config = RuntimeConfigStore()   # module-level singleton

Validation (range checks, allowed keys) lives in the unified agent's manage_config tool handler, not in the store itself. The store is a pure read/write layer — keeping validation out of it means the store stays trivially testable and the business rules are colocated with the tool definition.

Section 4

Software Engineering Best Practices Demonstrated

1. Separation of Concerns

Every module has exactly one reason to change. If the scraping website restructures its HTML, only scraper.py needs updating. If the fee threshold logic changes, only filters.py changes. If Gmail changes its SMTP settings, only notifier.py changes.

✅ Why it matters

Without it: A single 500-line bot.py mixes HTTP calls, HTML parsing, business rules, email sending, and logging. Changing the email format requires reading and understanding the entire file. A bug in scraping breaks email tests. New developers can't understand where to look for anything.

With it: Each module is small enough to read in 5 minutes. Tests are isolated. Debugging is linear — "the email isn't sending" → open notifier.py.

2. DRY — Don't Repeat Yourself

The parsing helpers (parse_min_fee, normalize_to_yyyymmdd, parse_weekday, parse_start_time) are defined once in filters.py and imported by calendar_client.py and integrations/telegram_bot.py. The GigFilterChain is instantiated twice (pre-filter and main filter) with different filters loaded, rather than copy-pasting filter logic.

3. SOLID Principles

PrincipleWhere applied
Single ResponsibilityEach module and class has one job: Scraper fetches HTML, FeeFilter checks fees, Notifier sends emails
Open/ClosedGigFilterChain is open to extension (add a new filter class), closed to modification (the chain code itself never changes)
Liskov SubstitutionFakeTransport and SMTPTransport are interchangeable anywhere a transport is expected
Interface SegregationFilters only expose __call__(gig) -> bool — no methods the chain doesn't need

4. Fail-Open Error Handling

Three external systems can fail independently: Google Maps API, Google Calendar API, and the scraping website. The design principle is: a transient external error must never silently cause a gig to be missed.

# CalendarFilter — if the API call fails, pass the gig through
except Exception:

    return False   # False from has_event_on_date means "not busy" → gig passes

5. Structured Logging with Context

Every log call carries structured context via the extra={} argument. This means the logs are queryable: find all runs where scraping took more than 2 seconds (jq 'select(.phase == "scrape" and .elapsed_ms > 2000)'). The run_id field correlates all messages from a single poll run.

    "pre_filter_passed": pre_filter_passed,
    "scraped": len(gig_list),
    "gig_errors": gig_errors,
    "elapsed_ms": int((time.perf_counter() - t0) * 1000),
})

6. Configuration Management with Feature Flags

Every filter can be disabled with a single environment variable: ENABLE_FEE_FILTER=false. This is invaluable during development (disable the postcode filter to avoid API charges) and debugging (disable the seen filter to re-test with already-seen gigs). Flags are additive — you never need to comment out code.

7. Testing Strategy

The test suite follows three patterns: unit tests for pure functions (parsing helpers), integration tests with mocked I/O (scraper tests with fake HTML, filter tests with mock Maps clients), and contract tests for the orchestrator (test_main.py verifies the three-phase flow with FakeTransport). The FakeTransport pattern is particularly valuable — it lets you assert on what emails would have been sent without a real SMTP server.

8. CI/CD Pipeline

Two GitHub Actions jobs run in parallel on every push. The lint job catches style, formatting, and type errors. The test job runs the full test suite. Failures in either block merging. Pre-commit hooks catch the most common issues before the code even reaches GitHub, reducing CI cycle time.

💡 First Principles: What is CI?

Continuous Integration (CI) is the practice of automatically running your quality checks every time code is pushed. Before CI, developers would work on separate branches for weeks, then spend days resolving conflicts and fixing bugs that accumulated undetected. CI surfaces problems within minutes of their introduction, while the context is fresh in the developer's mind. The "continuous" part means you integrate small changes frequently, rather than large changes rarely.

Section 5

Python Best Practices & Idioms

1. Callable Objects via __call__

Each filter is an object, not a function. The __call__ dunder method makes it callable. This combines the power of functions (can be passed around, stored in lists) with the power of objects (can hold state and configuration).

# A plain function can't remember min_fee between calls:
def fee_filter(gig): return gig.fee >= ???

# A callable object remembers its configuration:
f = FeeFilter(min_fee=80)   # configure once
f(gig1)                       # call many times
f(gig2)

2. The Context Manager Protocol

with Scraper() as scraper:
    html = scraper.fetch(url)
    # scraper.session.close() called automatically on exit

The with statement guarantees that __exit__ is called even if an exception is raised. This is the Pythonic equivalent of try/finally — it ensures resources (HTTP connections, file handles, database connections) are always released.

3. Type Hints and Union Types

def parse_min_fee(fee_str: str | None) -> float | None: ...

Python's type hints are documentation that mypy can verify. str | None (Python 3.10+ syntax) explicitly documents that the function can receive and return nothing. This turns a runtime surprise ("why is this None?") into a compile-time error.

4. List Comprehensions

# Instead of:
valid = []
for gig in gigs:
    if self.is_valid(gig):
        valid.append(gig)

# Pythonic:
valid = [gig for gig in gigs if self.is_valid(gig)]

List comprehensions are faster than equivalent loops (the list is built internally without repeated append calls) and more readable when kept to one logical operation.

5. Set Comprehensions

# Convert a list of CSV rows into a set of URLs — O(1) lookup
return {row["link"] for row in reader if row.get("link")}

6. contextvars for Implicit Context Propagation

Python's contextvars module provides context variables that are automatically scoped to the current execution context. The run_id is set once at the top of main() and is readable anywhere in the call stack for the duration of that run — without being passed as a parameter to every function. This is the correct, thread-safe alternative to a global variable.

7. frozenset for Immutable Lookup Tables

_STDLIB_FIELDS = frozenset({"args", "asctime", "created", ...})

A frozenset is a set that cannot be modified after creation. Using one for _STDLIB_FIELDS (the set of built-in logging record fields to exclude from JSON output) signals intent: this is a constant, not a variable. It also makes the membership check slightly faster and thread-safe.

8. f-strings with !r Formatting

logger.info(f"PostcodeFilter(home={self.home_postcode!r})")

The !r conversion calls repr() on the value, which adds quotes around strings and shows None literally. This is essential in log messages because repr("SW1A 1AA") produces 'SW1A 1AA' (with quotes) while str("SW1A 1AA") produces SW1A 1AA (without) — the quotes make it unambiguous that the value is a string.

9. Dependency Injection via Constructor

class PostcodeFilter:
    def __init__(self, home_postcode, api_key, max_minutes=45,
                 _client=None):  # injectable for testing
        self._client = _client or googlemaps.Client(key=api_key)

By accepting an optional _client parameter, tests can inject a mock without any monkey-patching of the module. This is pure dependency injection — the class doesn't create its own dependencies, they are handed to it.

10. Rotating File Handler

file_handler = logging.handlers.RotatingFileHandler(
    log_file,
    maxBytes=5 * 1024 * 1024,  # rotate at 5 MB
    backupCount=3,               # keep 4 files total
)

Without rotation, a long-running bot would grow a log file indefinitely. The RotatingFileHandler caps the file at 5 MB and keeps up to 3 backups (gigs.log, gigs.log.1, gigs.log.2, gigs.log.3). This is production-standard log management using only Python's standard library.

Section 6

Performance Optimizations — Deep Dive

OrganistBot is an I/O-bound application: the bottleneck is network latency, not CPU computation. Each of the five optimisations below targets a specific I/O cost. Understanding why they work requires understanding how the computer spends its time.

💡 First Principles: I/O-bound vs CPU-bound

A CPU-bound program spends most of its time computing (sorting a million numbers, rendering a video). Making it faster means using faster algorithms or more CPU cores. An I/O-bound program spends most of its time waiting — for a disk read, a network response, a database query. The CPU is idle during that wait. For I/O-bound programs, the key is to reduce the number of waits and make each wait shorter. All five optimisations below do exactly that.

💡 First Principles: What is a TLS handshake?

When your browser (or a Python program) connects to an HTTPS website, it must first negotiate an encrypted channel. This involves several round-trips: the client says hello, the server presents its certificate, both sides agree on encryption keys. This negotiation — the TLS handshake — typically takes 200–500ms and happens before any actual data is transferred. Connection pooling keeps the established encrypted channel open so subsequent requests skip the handshake entirely.

def _travel_times(self, postcode: str) -> dict[str, int | None]:
    if postcode not in self._cache:
        # Only call the API if we haven't seen this postcode before
        self._cache[postcode] = {
            mode: self._query(postcode, mode) for mode in self.MODES
        }
    return self._cache[postcode]  # O(1) dict lookup on subsequent calls

The ordering of filters in main.py is intentional across two separate chains. The Phase 1 pre-filter (cheapest-to-most-expensive, runs before the detail-page HTTP fetch): SeenFilter (O(1) set lookup) → FeeFilter (pure arithmetic) → SundayTimeFilter (pure arithmetic) → BookedDateFilter (set lookup) → CalendarFilter (API call). The Phase 2 filter chain handles fields only available after fetching the detail page: FeeFilter → SundayTimeFilter → BlacklistFilter (set lookup) → BookedDateFilter → PostcodeFilter (API call). CalendarFilter sits in Phase 1 because it only needs the gig's date — no detail-page fetch required. PostcodeFilter stays in Phase 2 because it needs the postcode, which only appears on the detail page.

Product Development Lessons

Requirements → Design → Implementation → Iteration

OrganistBot is a textbook example of iterative product development visible in the code itself. The core requirement was simple: tell me when a suitable gig appears. The first implementation needed only a fee filter and a seen-gigs store. The codebase shows the iterations clearly: the pre-filter chain (added after observing slow runs with many listings), the Telegram booking flow (added after the email-only workflow proved cumbersome), the CalendarFilter (added after double-booking became a real problem).

The feature flags (enable_fee_filter, enable_postcode_filter, etc.) are a direct record of this iteration — they were added as each feature was introduced, so existing deployments could opt in gradually without breaking changes.

Concepts Glossary — First Principles Reference

API (Application Programming Interface)
A defined contract through which two pieces of software communicate. Like a restaurant menu — you choose from the listed options and receive a defined response; you don't need to know how the kitchen works. In this project, the Google Calendar API, Google Maps Distance Matrix API, and Telegram Bot API are all external APIs accessed over HTTP using JSON.
BeautifulSoup
A Python library for parsing HTML and XML documents. It turns a raw HTML string into a navigable tree of Python objects so you can ask questions like "find the first <h3> tag with class 'organisation'". It is tolerant of malformed HTML, which is important when scraping real-world websites that rarely produce perfectly valid markup.
CI/CD (Continuous Integration / Continuous Delivery)
CI is the practice of automatically running quality checks (tests, linters, type checkers) every time code is committed. CD extends this by automatically deploying passing code to production. OrganistBot implements CI via GitHub Actions but not CD, since it runs locally on a Mac. The goal of CI is to detect and fix problems when they are small and fresh, not after weeks of accumulation.
Connection Pooling
Keeping a set of established network connections alive and reusing them for subsequent requests. Establishing a TCP connection and negotiating TLS encryption takes 200–500ms. By pooling connections, that cost is paid once, not on every request. The requests.Session with an HTTPAdapter in this project implements connection pooling for the scraper's HTTP requests.
Context Variable (contextvars.ContextVar)
A variable whose value is scoped to the current execution context (thread, async task, or coroutine). Unlike a global variable (one value shared by all code), a ContextVar can have different values in different concurrent tasks. In this project, _run_id is a ContextVar — each poll run sets it once, and every logging call within that run automatically reads the correct run_id without it being passed as a parameter.
Dataclass
A Python class decorated with @dataclass that auto-generates boilerplate: __init__, __repr__, and __eq__. It is a concise way to define data-holding classes. Unlike a plain dict, a dataclass provides attribute access (gig.fee instead of gig["fee"]), type annotation, and a human-readable repr for debugging.
Dependency Injection
A design pattern where an object's dependencies are provided from outside, rather than created internally. Instead of self.client = googlemaps.Client(key=api_key) always, the code accepts an optional _client=None parameter. Tests inject a mock; production uses the real client. This makes testing possible without patching global module state, and makes the class's dependencies explicit and inspectable.
Duck Typing
Python's approach to polymorphism: "if it walks like a duck and quacks like a duck, it's a duck." Rather than requiring objects to formally declare they implement an interface, Python checks whether the object has the required method when that method is called. SMTPTransport and FakeTransport both have a send() method — either works wherever a transport is expected, without any formal interface declaration.
Environment Variables
Key/value pairs inherited by a process from the operating system. They are the standard Unix mechanism for passing configuration (especially secrets) to programs without embedding values in source code. A .env file is a developer convenience — it contains those key/value pairs and is loaded into the process environment by python-dotenv at startup. The .env file is listed in .gitignore and never committed to version control.
Exponential Backoff
A retry strategy where the wait time between attempts doubles with each failure: 1s, 2s, 4s, 8s. It is the standard approach for handling transient network errors because it reduces load on an already-stressed server and spreads out retry traffic. The tenacity library implements this with the wait_exponential() parameter. A maximum wait cap (15s in this project) prevents indefinitely long waits.
Fail-Open vs Fail-Closed
When an external dependency fails, "fail-open" means allowing the operation to proceed as if the dependency reported a permissive result; "fail-closed" means blocking the operation. For a notification bot, fail-open is usually correct: if the Google Maps API fails, passing the gig through (possible false positive) is better than silently dropping it (definite false negative). The choice depends on whether false positives or false negatives are more harmful.
Feature Flag
A configuration switch that enables or disables a feature at runtime without deploying new code. In OrganistBot, each filter has a corresponding enable_*_filter boolean in Settings. Setting ENABLE_POSTCODE_FILTER=false in .env disables that filter instantly. Feature flags allow gradual rollouts, A/B testing, and fast disablement of broken features in production.
GitHub Actions
GitHub's built-in CI/CD platform. Workflows are defined in YAML files under .github/workflows/. A workflow runs on a GitHub-managed virtual machine (called a "runner") when triggered by events like a git push. Each workflow consists of jobs, each job consists of steps, and each step either runs a shell command or invokes a pre-built "action" from the GitHub marketplace.
HTTP Keep-Alive
An HTTP feature that keeps a TCP connection open after a request completes, allowing subsequent requests to reuse it. Without keep-alive, every HTTP request requires a new TCP three-way handshake (~50ms) and, for HTTPS, a TLS handshake (~300ms). The requests.Session uses keep-alive by default. The HTTPAdapter with pool_maxsize=10 extends this by maintaining a pool of up to 10 such connections.
Idempotency
An operation is idempotent if performing it multiple times has the same effect as performing it once. The setup_logging() function is idempotent — calling it twice only sets up handlers once. The save_seen_gigs() function is idempotent — writing the same URLs twice doesn't create duplicates because the seen-gigs set deduplicates them on load. Idempotency is valuable because it makes functions safe to retry.
Jinja2
A template engine for Python. Templates are text files (HTML in this project) with special syntax for placeholders ({{ variable }}), loops ({% for item in list %}), and conditionals ({% if condition %}). At render time, Jinja2 fills in the placeholders with real data. This separates presentation (HTML/CSS) from logic (Python), making each independently editable.
Memoisation
Caching the result of a function call so that identical future calls return the cached result immediately. The function computes its result once, stores it in a dictionary keyed by the inputs, and returns the cached value on subsequent identical calls. The PostcodeFilter._cache is a manual memoisation of the Google Maps API call. Python's functools.lru_cache decorator automates memoisation for pure functions.
Mypy
Python's optional static type checker. It reads your type annotations (def f(x: int) -> str) and verifies them without running the code. Mypy catches type mismatches, wrong argument types, and possible None dereferences before they cause runtime errors. It is "optional" in the sense that Python runs fine without annotations — mypy is a separate tool you run as part of CI.
Pipeline Architecture
An architectural pattern where data flows through a sequence of processing stages, each transforming it into the input for the next. OrganistBot's Scrape → Filter → Notify pipeline is a classic example. Pipelines are composable (stages can be added/removed), testable (each stage can be tested with fake inputs), and observable (each stage can be logged independently).
Pre-commit Hook
A script that runs automatically when you attempt a git commit. If the script exits with a non-zero code (failure), the commit is aborted. Pre-commit hooks enforce quality gates at the source — before code reaches the remote repository or CI. The pre-commit framework manages hook installation and runs hooks in isolated virtual environments to avoid conflicts with the project's own dependencies.
Pydantic / pydantic-settings
Pydantic is a Python library for data validation using type annotations. You declare a class with typed fields, and Pydantic validates and coerces input data to those types — raising clear errors if validation fails. pydantic-settings extends this for configuration: it reads values from environment variables and .env files, validates their types, and raises errors at startup if required settings are missing.
pytest
Python's de-facto standard testing framework. Tests are plain functions starting with test_; pytest discovers and runs them automatically. The mocker fixture (from pytest-mock) provides a clean interface for creating mocks. The tmp_path fixture provides a temporary directory for file-based tests. pytest-asyncio adds support for async test functions.
Regex (Regular Expression)
A mini-language for describing text patterns. r"£?\s*([0-9]+(?:\.[0-9]{1,2})?)" means: optional £, optional whitespace, then capture a number with up to 2 decimal places. Python's re module compiles and evaluates these patterns. Regex is the right tool for flexible text extraction from semi-structured data like web-scraped fee strings that vary in format.
Rotating Log File
A log file that is automatically archived and replaced when it reaches a size limit. Python's RotatingFileHandler renames gigs.log to gigs.log.1 when it hits 5 MB, then starts a fresh gigs.log. Old backups are kept up to a backupCount limit. Without rotation, a long-running bot's log file would grow without bound, eventually filling the disk.
Web Scraping
Programmatically downloading web pages and extracting structured data from their HTML. Since most websites are designed for human readers, scraping requires understanding the HTML structure and writing code to navigate it. The main challenges are: sites change their HTML structure without notice, they may block automated requests, and the data is often inconsistently formatted (hence the many parsing helpers in filters.py).
SMTP (Simple Mail Transfer Protocol)
The protocol used to send email between servers. Python's smtplib.SMTP_SSL connects to Gmail's SMTP server on port 465, authenticates with the sender's credentials, and delivers the email message. The email itself is structured as a MIME (Multipurpose Internet Mail Extensions) document — a format that supports HTML bodies, attachments, CC lists, and other email features.
Type Hints
Annotations in Python source code that specify the expected types of variables, parameters, and return values. They are not enforced at runtime (Python ignores them when executing) but are read by type checkers like mypy and displayed by IDEs. They serve as machine-verifiable documentation: def parse_min_fee(fee_str: str | None) -> float | None tells the reader exactly what this function accepts and returns.
UUID (Universally Unique Identifier)
A 128-bit number generated to be statistically unique across all systems and all time, without coordination. OrganistBot generates a random UUID at the start of each poll run and uses the first 8 hex characters as a run_id to correlate log messages. uuid.uuid4().hex[:8] produces an 8-character hex string like a1b2c3d4 that is unique enough for log correlation purposes.
